# 12. 安装/发布/自动更新（GitHub Actions + 更新 UX）

本文档定义 Windows 桌面端的打包产物、GitHub Actions 发布策略、版本通道策略（可选）、自动更新（Auto Update）流程与失败回退、以及签名/密钥管理原则（密钥不入库）。不包含任何实现代码与 CI YAML。

## 1) 发布目标与产物形态（Windows installer：NSIS；版本号规则；portable 可选）

### 1.1 发布目标

- MUST: 面向 Windows 10/11 发布可安装版本（NSIS 安装包），支持用户正常安装/卸载、覆盖安装与自动更新。
- SHOULD: 提供便携版（portable）作为可选产物，用于免安装使用与排障，但需明确：便携版不作为自动更新的默认通道。

### 1.2 产物形态（Windows-only）

- MUST: NSIS 安装包（installer），作为主发布产物。
  - 典型文件形态：`*-Setup-<version>.exe` 或 `*-Installer-<version>.exe`（名称在本项目内固定一种即可）。
  - 约束关键字：必须使用 `NSIS` 目标（electron-builder）。
- SHOULD: 便携版（portable）
  - 典型文件形态：`*-Portable-<version>.zip`（解压即用）。
  - MUST: 便携版的下载入口只能在 `Releases` 附件中提供；应用内“自动更新”不应指向便携版。
- MUST: 校验文件（hash）
  - 典型文件形态：`SHA256SUMS.txt`（或 `*.sha256`）。
  - MUST: 所有发布附件都必须在同一份校验文件中列出，并在发布说明中提示用户可用 `sha256` 校验。

### 1.3 版本号规则（Windows-only）

- MUST: 语义化版本（SemVer）：`MAJOR.MINOR.PATCH`。
- MUST: 正式发布使用不带后缀的版本号（例如 `0.3.0`）。
- SHOULD: 若启用通道（见 3.2），预发布版本使用 SemVer 预发布标记：
  - `0.3.0-beta.1`（beta 通道）
  - `0.3.0-rc.1`（候选）
  - MUST: 自动更新只在同一通道内推进（避免稳定版误升级到 beta）。

## 2) GitHub Actions 发布流程（只写策略与步骤，不写具体 yaml 代码）

目标：通过 `GitHub Actions` 在 Windows Runner 上完成构建、打包、签名（可选但推荐）、生成 hash，并将产物上传到 `Releases`。

### 2.1 触发条件（建议组合）

- MUST: `push tag` 触发发布（例如 `v0.3.0`）。
- SHOULD: `workflow_dispatch` 触发手动发布（用于热修、重发、或复现打包）。
- SHOULD: PR/branch 仅跑构建与测试，不发布到 `Releases`（避免未审计产物流出）。

### 2.2 构建矩阵（win）

- MUST: 仅 Windows（例如 `windows-latest`）。
- SHOULD: Node 版本固定（例如 `18` 或 `20`），避免打包产物不可复现。
- SHOULD: 启用依赖缓存（npm/pnpm/yarn 依实际项目），提升构建稳定性。

### 2.3 发布步骤（策略化描述）

- MUST: 检出代码（含 tag），安装依赖，执行基础校验（lint/test/typecheck 以项目既有脚本为准）。
- MUST: 构建应用（renderer/main/preload 等）并执行打包。
- MUST: 生成 NSIS 安装包（`NSIS`），并确保产物版本号与 tag 一致。
- SHOULD: 产物签名（见第 4 章），签名对象至少包含安装器 exe 与最终 app 可执行文件。
- MUST: 生成 `SHA-256` 校验（hash 校验），产出单一校验文件并纳入 Release 附件。
- MUST: 创建/更新 GitHub `Releases`：
  - 标题：`v<version>`
  - 说明：包含变更摘要、已知问题、以及校验方式提示。
  - 附件：安装包、（可选）便携版、校验文件（hash）。

### 2.4 产物命名规范（可验收）

- MUST: 文件名包含：应用名 + 版本号。
- SHOULD: 若启用通道，文件名包含通道标记（例如 `beta`），避免用户误下。
- MUST: 同一个版本在 `Releases` 中的附件命名稳定且可预测（便于自动更新与支持人员定位）。

## 3) 自动更新策略（Auto Update）

约束：Windows 自动更新依赖 NSIS（见 `03-windows-integration.md` 第 5 章对“自动更新（Windows/NSIS）”的约束）。本节仅定义策略与用户体验，不定义实现。

### 3.1 检测频率（启动/定时/手动）

- MUST: 启动检测：应用启动后（且 UI 可响应）进行一次“轻量检查”。
- SHOULD: 定时检测：每 `6~12` 小时检查一次（避免频繁打断与浪费带宽）。
- MUST: 手动检测：设置页提供“检查更新”按钮，且可在任意时间触发。
- MUST: 所有检测都必须可被用户取消/忽略，不得阻断编辑与同步。

### 3.2 版本通道策略（可选，但一旦启用必须完整）

- OPTIONAL: 支持通道：`stable` / `beta`（最小两档即可）。
- MUST: 默认通道为 `stable`。
- MUST: 通道信息对用户可见且可切换（设置页），切换后下次检测生效。
- MUST: 同一通道内更新：
  - stable 只升级到 stable（不跨到 beta）。
  - beta 可升级到更高 beta/rc，且可允许用户手动回到 stable（行为必须明确提示）。

### 3.3 后台下载与提示（不打断编辑）

- MUST: 下载在后台进行，且不应抢占焦点、弹阻塞对话框或强制导航。
- MUST: 下载过程中：
  - 应用保持可编辑（笔记输入、拖拽、同步等不受影响）。
  - 仅通过“非阻塞提示”反馈进度（设置页状态行、托盘提示或应用内轻提示）。
- SHOULD: 对网络异常（离线、代理、超时）做友好提示，并提供“重试”动作。

### 3.4 重启安装 / 延后安装

- MUST: 下载完成后，不自动重启；必须由用户明确触发“安装更新”。
- MUST: 支持“延后安装”：
  - 用户可选择“稍后再说”，更新保持待安装状态。
  - 下次启动或定时检测时，仅提醒“可安装”，不重复下载。
- SHOULD: 当存在未保存编辑时：
  - MUST: 安装前提示用户保存。
  - SHOULD: 若应用具备自动保存机制，则提示可以更轻量，但仍要避免造成数据丢失的观感。

### 3.5 失败回退与用户动作（下载失败/校验失败/安装失败）

原则：任何失败都不应破坏当前可用版本；优先保证“继续使用当前版本 + 可重试 + 可手动下载”。

- MUST: 下载失败（网络/权限/磁盘空间不足）
  - MUST: 状态变为失败并给出原因摘要。
  - MUST: 提供动作：`重试下载`。
  - SHOULD: 提供动作：`打开 Releases`（跳转到对应版本的 GitHub Releases 页面，便于手动下载）。
- MUST: 校验失败（hash mismatch / 签名校验失败）
  - MUST: 视为不可信更新，丢弃已下载文件并提示用户。
  - MUST: 提供动作：`重新下载`。
  - SHOULD: 提供动作：`复制校验信息`（用于反馈/工单）。
- MUST: 安装失败（启动安装器失败/安装器返回错误/重启失败）
  - MUST: 保持当前版本继续运行。
  - MUST: 提供动作：`重试安装` 或 `打开 Releases`。
  - SHOULD: 引导用户查看日志（本项目已有日志规范则引用对应文档；此处不新增实现）。

## 4) 签名与密钥管理（证书/密钥不入库；CI secrets 注入；dev 不强制）

### 4.1 基本原则（强制）

- MUST: 证书/私钥/口令等敏感材料一律 `密钥不入库`。
- MUST: CI 中的签名材料只能通过 GitHub `secret`（或等价的机密管理）注入。
- MUST: 任何签名材料不得出现在：仓库文件、构建产物 commit、PR 日志输出（避免泄漏）。

### 4.2 本地开发 vs CI 的差异（应解释清楚）

- SHOULD: 本地开发（dev）不强制签名：
  - 允许未签名的调试/本地构建，重点是开发效率。
  - 但需要明确：未签名构建可能触发 Windows SmartScreen 警告，属于预期。
- MUST: CI 发布（tag/Release）必须执行签名（若已具备证书），以降低误报与提升可信度。

### 4.3 CI 注入方式（策略，不写脚本）

- MUST: 将签名 `证书` 以二进制安全的方式存入 GitHub `secret`（例如 base64 字符串），在构建时写入临时文件并在 job 结束后销毁。
- MUST: 签名口令/别名等也通过 `secret` 注入。
- MUST: 严格控制输出：禁止在日志中打印证书内容、base64 串、或任何可复用的签名材料。

## 5) 与 Windows 通知/AUMID 的关系：electron-builder appId 必须稳定（引用 learnings）

来自 `03-windows-integration.md` 的关键 learnings（Windows-only）：

- MUST: Windows 通知（Toast）要可靠展示，需要 Start Menu shortcut 与稳定的 AUMID 前置条件。
- MUST: electron-builder 的 `appId` 会作为 Windows 的 Application User Model ID（AUMID）使用，且强烈建议显式设置。
- MUST: `appId` 必须保持稳定（不要随版本/通道频繁变化），否则会导致：
  - 通知聚合/归属异常（系统把它当成不同应用）。
  - 与“检查更新 / 下载完成 / 需要重启安装”等更新提示的通知链路不稳定（需要降级到应用内提示或托盘气泡）。

验收点：发布与自动更新流程中使用的 `appId` 固定且可追溯；任何变更都必须作为破坏性变更评审。

## 6) 更新 UX 最小规范（最多 3 个状态；每个状态 1 行提示 + 1 个动作）

约束：更新 UI 必须“非阻塞”“不打断编辑”。更新入口建议放在设置页（同时托盘可提供只读状态）。

仅允许以下 3 个状态（其他细节信息如版本号/大小可放在同一行里，但不新增状态）：

1) 检查中
- 提示（1 行）：`正在检查更新…`
- 动作（1 个）：`取消`

2) 下载中
- 提示（1 行）：`正在后台下载更新（xx%）…`
- 动作（1 个）：`暂停/取消下载`（二选一，保持全局一致）

3) 可安装
- 提示（1 行）：`更新已准备好，重启后安装（vX.Y.Z）。`
- 动作（1 个）：`安装更新（重启）`（必须二次确认可在点击后出现，但不新增状态行）

失败呈现规则（不新增状态）：当失败发生时，复用当前状态行的提示文案，将其替换为“失败原因 + 下一步动作引导”，动作仍保持 1 个（优先 `重试`；必要时 `打开 Releases`）。
